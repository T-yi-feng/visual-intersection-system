## 1. 项目背景

### 1.1 问题定义

城市交通交叉口是路网拥堵的核心节点。现有的交通监控系统大多仅提供视频回放功能，缺乏对交叉口内部车辆交互行为的定量分析能力。具体而言，以下问题亟待解决：

1. **冲突检测效率低**：传统基于成对几何的冲突检测方法（TTC/PET）复杂度为 O(N²)，在密集交通流（30+ 辆/帧）场景下性能显著下降。
2. **拥堵可解释性不足**：现有拥堵指数多基于宏观断面数据（占有率、流量），无法定位"哪辆车导致了拥堵"。
3. **单车归因缺失**：缺乏量化每辆车对拥堵贡献度的计算方法，导致信号配时优化缺乏微观层面的数据支撑。
4. **遮挡导致 ID 跳变**：车辆被大型车（公交车、卡车）遮挡后，跟踪器分配新 ID，影响轨迹连续性和归因准确性。

### 1.2 现有方案局限

| 方案 | 局限 |
|------|------|
| 基于 TTC 的成对冲突检测 | O(N²) 复杂度，难以 GPU 加速，在密集场景下计算延迟高 |
| 宏观交通指数（TCI/API） | 基于断面数据，无法定位单车影响 |
| 传统目标跟踪（SORT/DeepSORT） | 遮挡后 ID 丢失，缺乏记忆恢复机制 |
| 基于图神经网络的归因 | 依赖标注数据训练，部署成本高，泛化性受限 |

### 1.3 本文方案

提出一种**基于方向场卷积的冲突检测与拥堵归因框架**，核心思路是将"车辆间潜在冲突"建模为方向性路经影响场的交叉，通过固定栅格上的场卷积运算（O(G²)）替代传统成对几何计算（O(N²)），实现 GPU 友好的实时冲突检测与单车归因。

### 3.2 创新详细说明

#### 创新 1：方向场卷积冲突检测

本系统最核心的创新。用 G×G 固定栅格上的场卷积运算替代传统 O(N²) 的成对几何冲突检测。

**传统方法**：
```
for i in range(N):
    for j in range(N):
        TTC_ij = compute_TTC(vehicle_i, vehicle_j)  # O(N²)
```

**本方法**：
```
O_k = scatter_to_grid(vehicles)          # O(N) 散布
R_k = cv2.filter2D(O_k, K_k)             # O(G²) 卷积，GPU加速
C = sum(R_a * R_b for (a,b) in pairs)    # O(K) 累加，K=24固定
```

**关键收益**：
- 复杂度从 O(N²) 降为 O(G²)，G=64 固定
- `cv2.filter2D` 硬件加速（NVIDIA 显卡上由 CUDA 执行）
- 不随车辆数增加而增加计算量

#### 创新 2：各向异性扇形核

传统各向异性核为等宽高斯（kernel 宽度不随距离变化）。本系统提出前向扇形展开核：

```python
# 传统等宽：
f_perp = exp(-0.5 * (perp / sigma_perp) ** 2)

# 本系统扇形：
if along >= 0:
    fan_sigma = sigma_perp * (1.0 + 0.6 * along / sigma_along)
f_perp = exp(-0.5 * (perp / fan_sigma) ** 2)
```

物理含义：车辆前方的影响范围呈扇形展开（类似车灯光束），更符合真实交通流中车辆对前方多车道的影响模式。

#### 创新 5：水滴传播因果溯源

提出基于稀疏矩阵迭代传播的拥堵根因定位算法：

- 每辆车携带初始"水滴"（x_i = 1.0）
- 通过冲突场连接构建邻接矩阵 A
- 水沿拥堵链逆流传播：x += α · (A^T @ x)
- 经过 10-15 轮迭代收敛
- 冲突热区的水汇聚量最大 → 定位为根因

与传统 PageRank 算法有相似之处，
但连接权重由冲突场动态决定，无需先验图结构。

## 5. 核心算法详解

### 5.1 方向场卷积冲突检测

**数学形式化**：

设 BEV 平面离散化为 G×G 栅格，每格边长 `cell_size = WorldSize / G`。

**散布**：
```
O(x,y) = Σ δ(x - x_i, y - y_i)        # 占用场
V(x,y) = Σ v_i · δ(x - x_i, y - y_i)  # 速度场
```

**方向分解（软分配）**：
对于方向箱 k（中心角度 θ_k = k × 30°），每辆车对其贡献：
```
O_k(x,y) = Σ w_i_k · δ(x - x_i, y - y_i)
w_i_k = exp(-Δθ² / 2σ²)               # 高斯权重
σ = 30°/3 = 10°
```

**各向异性核**：
```
K_k(u,v) = exp(-along²/2σ_a²) × exp(-perp²/2σ_p²)
其中:
  along = u·cos(θ_k) - v·sin(θ_k)   # 沿方向投影
  perp  = u·sin(θ_k) + v·cos(θ_k)   # 垂直方向投影
  σ_a(v) = σ_a0 × (1 + α·v/v_ref)    # 速度感知
  σ_p(along) = σ_p0 × (1 + 0.6·along/σ_a0)  # 扇形展开
```

**路径影响场**：
```
R_k = O_k ⊛ K_k    # cv2.filter2D
```

**冲突场**：
```
C(x,y) = Σ_{(a,b)∈Pairs} R_a(x,y) · R_b(x,y)
Pairs = {(0,6),...,(5,11),        # 对向 180°
         (0,3),...,(5,8),          # 正交 ~90°
         (0,0),...,(11,11)}        # 同向跟驰 0°
```

**单车归因**：
```
Influence_i = R_{k_i}(x_i, y_i) · Σ_{k'∈Conflicts(k_i)} R_{k'}(x_i, y_i)
```

### 5.2 拥堵指数 Phi

```
Φ = w_ρ · min(1, N/N_sat) + w_v · max(0, 1 - v̄/v_ref)
```

双通道设计：
- **密度通道**：N/N_sat 反映空间占用率，饱和上限为 N_sat=40
- **速度通道**：1 - v̄/v_ref 反映速度衰减率，v_ref=5.0m/s ≈ 18km/h

权重 w_ρ=0.4, w_v=0.6 表明速度衰减比密度对拥堵判定更重要，
符合交通工程中"速度是拥堵最敏感的指标"的经验。

### 5.3 水滴传播因果溯源

**邻接矩阵构建**：
```
A_ij = C(中点_ij) × exp(-dist²/2σ²) 
条件: j 在 i 前方（方向投影 > 0） 且 中点有冲突（C > 0）
```

**迭代传播**：
```
x⁽⁰⁾ = [1, 1, ..., 1]
x⁽ᵗ⁺¹⁾ = x⁽ᵗ⁾ + α · (A_norm^T · x⁽ᵗ⁾)
A_norm = A / row_sum(A)        # 行归一化
```

**冲突场加权**：
```
RootCause_i = x_i · min(10 · C(x_i, y_i), 1)
```

### 5.4 空间-外观记忆匹配

**特征**: HSV 颜色直方图 (32×32 bin)  
**匹配评分**: IoU × 0.4 + 颜色相似度 × 0.3 + 运动连续性 × 0.3  
**记忆窗口**: 5 帧过期  
**恢复内容**: track_id + heading_deg（遮挡前的行驶方向）

### 7.2 消融实验

**目的**：验证单车归因的准确性。逐级移除归因分数最高的车辆后观测冲突场降幅。

| K 级 | 移除车辆 | 冲突场降幅 | Phi 降幅 |
|------|---------|-----------|---------|
| K=1 | Top-1 归因车 | 34% | 0.12 |
| K=2 | Top-2 | 51% | 0.23 |
| K=3 | Top-3 | 68% | 0.35 |

Top-3 车辆移除后冲突场下降 68%，说明归因主要集中在少数关键车辆上，与真实交通拥堵由少数车引发的经验一致。

### 7.3 核参数敏感性

| sigma_along | 前向有效范围 | 冲突场峰值 | 召回率 |
|-------------|-------------|-----------|--------|
| 2.0 | ~4.7m | 0.042 | 86% |
| **3.0** | **~7.0m** | **0.038** | **91%** |
| 4.0 | ~9.4m | 0.035 | 93% |
| 5.0 | ~11.7m | 0.032 | 94% |

sigma_along=3.0 在冲突场峰值与召回率间取得最佳平衡。过大的 sigma 导致影响场过度扩散（峰值下降），过小则遗漏远距离跟驰冲突。

### 7.4 栅格分辨率影响

| G | 每格大小 | 冲突检测 FPS | 空间分辨率 |
|---|---------|-------------|-----------|
| 32 | 1.56m | >200 | 粗，小车间距不可分辨 |
| **64** | **0.78m** | **~150** | **适中** |
| 128 | 0.39m | ~40 | 精细，但计算量4x |

G=64 为推荐设置。

| 7 | 同向车队无冲突 | 第 12 周 | 缺少自配对 | 跟驰无热力 | 添加 (k,k) 自配对 |
| 8 | 记忆恢复方向丢失 | 第 14 周 | engine 覆写 heading | 箭头默认方向 | meta.get 保留记忆值 |

## 9. 关键决策记录

### 9.1 为什么选择 YOLO11 而非 YOLOv8？

YOLO11 在同等参数量下 mAP 比 YOLOv8 高约 2.3%，推理速度略快。对于监控场景，精度的边际提升比速度更重要。

### 9.2 为什么选择 ByteTrack 而非 BotSORT？

ByteTrack 的低置信度检测保留策略更适合交通场景——遮挡后重新出现的目标往往置信度较低。BotSORT 的外观特征匹配在车辆颜色相近时效果有限。

### 9.3 为什么选择 12 个方向箱？

12 箱 × 30° 的设置在方向精度与计算量之间取得了平衡。8 箱（45°）对斜交路口的冲突检测精度不足；16 箱（22.5°）增加了卷积核数量但精度提升不明显。

### 9.4 为什么选择同向跟驰自配对？

最初设计仅包含对向和正交冲突对，但实际交通中跟驰冲突（追尾）是最常见的事故类型。添加 (k,k) 自配对后，C += R_k² 自然地捕获了同向跟驰的冲突强度。

### 9.5 为什么舍弃了位置信号？

水滴传播算法最初包含一个跨方向的位置信号（pos_score），用于在全停排队时通过位置梯度定位根因。但测试发现该信号会将不同方向车辆的位置投影混在一起归一化，导致结果没有物理意义。最终完全依赖传播信号 + 冲突场加权。

## 10. 不足与改进方向

### 10.1 当前局限

| 局限 | 影响 | 改进方向 |
|------|------|---------|
| 仅支持单摄像头 | 无法覆盖大面积路口 | 多视角融合 |
| 无车道线识别 | 无法区分合法/违章行驶 | 集成车道线检测 |
| 无信号灯状态 | 无法判断信号灯影响 | 信号灯识别 |
| G=64 固定栅格 | 大路口分辨率不足 | 自适应网格 |
| 视频离线处理 | 不支持实时流 | RTSP 输入 |
| 核参数固定 | 无法自适应交通流状态 | 动态参数调整 |

### 10.2 改进方向

**算法层面**：
- 多摄像头融合：多个摄像头覆盖同一路口的不同区域，统一到同一世界坐标系
- 自适应栅格分辨率：根据车辆密度动态调整 G
- 时序冲突预测：利用 LSTM 预测未来 5-10 秒的冲突趋势
- 信号灯联动：结合红绿灯相位信息分析拥堵成因

**工程层面**：
- TensorRT/ONNX 模型导出优化，提升推理速度
- 边缘端部署（Jetson Orin）
- 实时 RTSP 流接入
- WebSocket 推流替代文件写入

**验证层面**：
- 大规模数据集标注（含拥堵事件标注）
- 与地面真实 TTC 的对比验证
- 多城市交叉口泛化测试

### 10.3 算力优化空间

当前主要瓶颈为目标检测（YOLO11m，~60ms/帧）。以下为可行的优化路径：

| 优化手段 | 预期加速 | 实现复杂度 | 说明 |
|---------|---------|-----------|------|
| 换 yolo11s | 2-3× | ★ | 模型缩小 60%，精度损失约 3% |
| TensorRT 导出 | 1.5-2× | ★★★ | FP16 推理，需模型转换 |
| stride=2 跳帧 | 2× | ★ | 跳过交替帧，效果基本不变 |
| 降低 imgsz (960) | 1.3× | ★ | 降低输入分辨率 |
| 多线程视频读取 | 1.2× | ★★ | 流水线并行 |
