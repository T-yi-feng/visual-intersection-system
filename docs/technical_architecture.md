## 1. 系统概述

### 1.1 项目定位

本系统是一个**基于计算机视觉的交通交叉口实时监控与分析平台**。它从固定监控摄像头获取视频流，自动检测并跟踪车辆，通过鸟瞰图（BEV, Bird's Eye View）透视变换将画面映射到世界坐标系，并利用创新的**方向场卷积算法**实时分析车辆间的交织冲突与拥堵程度。

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| **车辆检测与跟踪** | YOLO11 + ByteTrack，支持 6 类车型（car/truck/bus/van/motorcycle/bicycle） |
| **BEV 透视变换** | 基于四角标定的单应性矩阵，将倾斜画面映射为俯视鸟瞰图 |
| **拥堵指数（Phi）** | 融合车辆密度与速度衰减的加权指数，实时量化拥堵风险 |
| **冲突检测** | 专利算法：方向场卷积，O(G²) 复杂度，GPU 友好，空间感知 |
| **单车归因** | 量化每辆车对当前拥堵/冲突的贡献度 |
| **消融验证** | 逐级移除高归因车辆，验证归因准确性 |
| **事件自动导出** | 拥堵事件自动截图、导出 CSV/JSON 摘要、消融结果 |

### 1.3 应用场景

- 城市交通管理部门的交叉口拥堵监控
- 交通信号配时优化（数据支撑）
- 交通规划与仿真验证
- 交通事故/异常事件的事后回溯分析

## 4. 核心模块详解

### 4.1 检测与跟踪 (`core/detector.py`)

```
Video Frame (BGR)
       │
       ▼
┌─────────────────┐
│  YOLO11 推理     │  ← 类别自适应置信度阈值
│  (car/truck/    │  ← 基于面积的标签修正
│   bus/van/      │     (小框→car, 大框→truck/bus)
│   motorcycle/   │
│   bicycle)      │
└────────┬────────┘
         │ detections[]
         ▼
┌─────────────────┐
│  ByteTrack      │  ← track_buffer=45
│  多目标跟踪      │  ← match_thresh=0.80
└────────┬────────┘
         │ tracked_vehicles[] (稳定 track_id)
         ▼
```

**设计要点**:
- **类别自适应阈值**: 小目标（motorcycle/bicycle）使用更高阈值以减少误检
- **面积标签修正**: 当 ByteTrack 的检测框面积与类别不匹配时（如大卡车被识别为 car），根据面积比例修正
- **跟踪缓冲区**: `track_buffer=45` 帧，允许短暂遮挡后恢复 ID

### 4.2 BEV 透视变换 (`core/bev_transform.py`)

```
像素坐标系 (倾斜监控画面)          世界坐标系 (鸟瞰俯视图)
                                  
    (0,0) ┌──────────┐              (0,0) ┌──────────┐
          │  ╱    ╲  │                    │          │
          │ ╱ 路口  ╲ │    ────►          │   路口   │
          │╱        ╲│   3×3 单应性        │          │
          └──────────┘   矩阵变换          └──────────┘
                                  (50m × 50m)
```

**核心流程**:
1. **标定**: `tools/calibrate_homography.py` 交互式拖拽 4 个角点，建立像素↔世界对应关系
2. **存储**: 4 对点坐标保存在 `configs/homography_<site>.json`
3. **计算**: `cv2.findHomography(img_pts, world_pts)` → 3×3 单应性矩阵 `H`
4. **单点映射**: `pixel_to_world(px, py)` → `(x_m, y_m)`（使用齐次坐标除法）
5. **全帧映射**: `cv2.warpPerspective(frame, H_bev, (W, H))` → BEV 鸟瞰图

**关键细节**:
- `H_bev` 复合矩阵 = 缩放 × 平移 × 单应性（确保 BEV 输出图像按世界长宽比正确缩放）
- 历史遗留问题修复：`H_bev` 的源四边形必须与 `h_mat` 使用同一组校准点（见 `docs/bev_fix_plan.md`）

### 4.3 运动分析 (`core/motion.py`)

```
车辆世界坐标轨迹 (deque)
       │
       ▼
┌──────────────────────┐
│  速度计算             │
│  v = Δd / Δt         │  ← sample_dt_s=0.5s 窗口
│  (世界坐标欧氏距离)    │
└──────────┬───────────┘

│  - moving:  v > v_min│

│  EMA 平滑             │

└──────────────────────┘
```

### 4.4 Phi 拥堵指数 (`core/phi.py`)

**公式**:

```
Phi = w_rho × min(1, N / N_sat)  +  w_v × max(0, 1 - v_avg / v_ref)

其中:
  N       = 当前场景车辆数
  N_sat   = 饱和车辆数 (默认 40)
  v_avg   = 车辆平均速度 (m/s)
  v_ref   = 参考自由流速度 (默认 5.0 m/s ≈ 18 km/h)
  w_rho   = 密度权重 (默认 0.4)
  w_v     = 速度权重 (默认 0.6)
```

**Phi ∈ [0, 1] 含义**:

| Phi 范围 | 状态 | 颜色 |
|----------|------|------|
| 0.00 – 0.30 | 畅通 | 绿色 |
| 0.30 – 0.55 | 轻度拥堵 | 黄色 |
| 0.55 – 0.75 | 中度拥堵 | 橙色 |
| 0.75 – 1.00 | 严重拥堵 | 红色 |

**事件追踪** (`PhiEventTracker`):
- **开始事件**: Phi 上穿 `phi_plot_threshold`（默认 0.70）
- **追踪峰值**: 持续记录 Phi 最大值及对应帧号
- **结束事件**: Phi 下穿阈值，触发事件导出流程

**参数热重载** (`RiskParamsReloader`):
- 定期检查 `traffic_risk_params.json` 文件修改时间
- 运行时可动态调整 N_sat、v_ref、权重等参数，无需重启

### 4.5 空间-外观记忆 (`core/memory.py`)

**解决问题**: 车辆被其他物体短暂遮挡后，ByteTrack 可能分配新 ID，导致轨迹断裂。

**方法**:
```
车辆消失时:
  1. 提取 HSV 颜色直方图 → 外观特征向量
  2. 记录最后位置 + 速度预测的运动方向
  3. 存入记忆库 (max N 辆车)

车辆重新出现时:
  1. 计算 IoU (位置匹配度)
  2. 计算颜色直方图距离 (外观匹配度)
  3. 计算运动连续性 (速度方向匹配度)
  4. 综合评分 → 超过阈值则恢复原 track_id
```

## 6. 核心算法：基于方向场卷积的冲突检测

> 这是本系统的**核心创新**，已申请专利（见 `docs/patent_spec_cn.md`）。  
> 该算法将传统 O(N²) 的成对几何冲突检测，转化为 O(G²) 的场卷积运算。

### 6.1 算法动机

传统方法对每对车辆计算 TTC（碰撞时间）/PET（后侵占时间）/相对角度，复杂度 O(N²)，在 40+ 辆车时性能骤降。本算法将"车辆间潜在冲突"建模为**方向性路径影响场的交叉**，利用 `cv2.filter2D` 的 GPU 加速卷积一次性完成。

### 6.2 算法流程

```
┌─────────────────────────────────────────────────────┐
│  Step 1: 散布到网格 (Scatter)                         │
│  ─────────────────────────────                       │
│  每辆车 → G×G 网格上的一个格子单元                      │
│  输出:                                                │
│    占用场 O[G][G]      (该格子有多少辆车)               │
│    速度场 V[G][G]      (该格子的平均速度)               │
│    方向场 Θ[G][G]      (该格子的主行驶方向, 0-360°)    │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│  Step 2: 方向分解 (Direction Discretization)          │
│  ───────────────────────────────────────             │
│  连续方向 0-360° → 离散化为 12 个方向箱 (每箱 30°)      │
│                                                       │
│    bin 0:  [  0°,  30°)     bin 6:  [180°, 210°)    │
│    bin 1:  [ 30°,  60°)     bin 7:  [210°, 240°)    │
│    ...                        ...                     │
│    bin 5:  [150°, 180°)     bin 11: [330°, 360°)    │
│                                                       │
│  输出: O_k[G][G]  (k=0..11), 该方向箱的二值占用场      │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│  Step 3: 各向异性核 (Anisotropic Gaussian Kernel)      │
│  ─────────────────────────────────────────           │
│  为每个方向箱 k 构建核 K_k:                             │
│                                                       │
│  沿行驶方向拉伸: σ_forward ≫ σ_side (如 3.0 vs 0.8)   │
│                                                       │
│    示意 (方向箱 0, 车辆向右):                           │
│    ┌───────┐                                         │
│    │   ░   │   ← 影响范围沿行驶方向延伸                 │
│    │  ███  │                                         │
│    │   ░   │                                         │
│    └───────┘                                         │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│  Step 4: 路径影响场 (Path Influence Field)             │
│  ─────────────────────────────────────               │
│  R_k = O_k ⊗ K_k     (cv2.filter2D 卷积)              │
│                                                       │
│  R_k(x,y) = 车辆群沿方向 k 的"势力范围"强度              │
│  强度从车道中心向两侧指数衰减                           │
│  从车辆位置向前方延伸（模拟行驶轨迹影响）                 │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│  Step 5: 冲突场 (Conflict Field)                      │
│  ─────────────────────────────                       │
│  C(x,y) = Σ R_a(x,y) × R_b(x,y)                      │
│           对所有预定义的冲突对 (a,b)                    │
│                                                       │
│  冲突对包括:                                           │
│    6 对对向冲突: (0,6), (1,7), ..., (5,11)             │
│    方向相差 180° (同一条路上的对向行驶)                  │
│                                                       │
│    6 对正交冲突: (0,3), (0,9), (1,4), (1,10), ...     │
│    方向相差约 90° (垂直路口方向)                        │
│                                                       │
│  冲突强度 ∝ R_a × R_b  (两条路径交叠则高冲突)          │
└────────────────────────┬────────────────────────────┘
                         ▼
┌─────────────────────────────────────────────────────┐
│  Step 6: 单车归因 (Per-Vehicle Attribution)           │
│  ───────────────────────────────────────             │
│  车辆 i 的归因分数:                                    │
│                                                       │
│  Influence_i = R_{k_i}(P_i) × Σ R_{k'}(P_i)          │
│                ───────────   ─────────────────        │
│                自身路径强度    位置冲突强度               │
│                                                       │
│  k_i: 车辆 i 所在的方向箱                              │
│  k': 与 k_i 冲突的方向箱                               │
│  P_i: 车辆 i 在网格中的位置                            │
│                                                       │
│  归一化后 → 归因百分比 (所有车加总 = 100%)              │
└─────────────────────────────────────────────────────┘
```

### 6.3 算法优势

| 维度 | 传统方法 (成对几何) | 本方法 (方向场卷积) |
|------|-------------------|-------------------|
| **复杂度** | O(N²) 逐对计算 | O(G²) 固定网格卷积 |
| **GPU 加速** | 难以并行化 | `cv2.filter2D` 天然 GPU 加速 |
| **空间感知** | 需额外建模 | 自动：路口中心权重高，边缘低 |
| **冲突对数量** | N×(N-1)/2 (40 车→780 对) | 固定 12 对方向箱 |
| **消融成本** | 需重算全部 | O(1)：只需重算一个方向箱 |
| **可解释性** | 离散的"冲突/不冲突" | 连续的热力图 + 单车归因分数 |

## 10. 扩展点与未来规划

### 10.1 当前架构限制

| 限制 | 影响 | 改进方向 |
|------|------|---------|
| 仅单摄像头 | 无法覆盖大面积路口 | 多摄像头融合（专利已预留接口） |
| 无车道线识别 | 无法区分合法/违章行驶 | 集成车道线检测（YOLOPv2 等） |
| 无信号灯状态 | 无法判断闯红灯行为 | 集成信号灯状态识别 |
| 固定 G×G 网格 | 大路口分辨率不足 | 自适应网格分辨率 |
| 离线视频处理 | 不支持实时流（RTSP） | 集成流媒体输入 |

### 10.2 可扩展方向

1. **多视角融合**: 多个摄像头覆盖同一路口的不同区域，统一到同一世界坐标系
2. **信号灯联动**: 结合红绿灯相位信息，分析信号配时对拥堵的影响
3. **长期趋势分析**: 积累数天/数周的 Phi 数据，生成拥堵规律报告
4. **边缘端部署**: 使用 TensorRT/ONNX 优化 YOLO，部署到 Jetson 等边缘设备
5. **API 化**: 将 PipelineEngine 封装为 HTTP/gRPC 服务，供上层平台调用

### 10.3 已知技术债务

- `analyze_phi.py` / `analyze_phi_v2.py` 为调试脚本，建议整合进 `tools/`
- 各站点 `runtime` 配置大量重复，可提取为 `default_runtime` + 站点增量覆盖
- 运行超长视频时（>2h）需关注 numpy 数组内存累积
- 单元测试覆盖核心算法但缺少端到端集成测试
